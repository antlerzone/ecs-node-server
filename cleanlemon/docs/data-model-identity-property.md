# Cleanlemons — data model: property, owner, client, integration auth

**Status:** design record for implementation (MySQL + product).

---

## 1) Property: separate Cleanlemons table (not Coliving-only)

- **Cleanlemons keeps its own property store** (e.g. `cln_property` / unit rows — exact names TBD).
- Reasons:
  - Cleanlemons operators serve **more than Coliving** (other software via webhook, native Cleanlemons clients, etc.).
  - Jobs, assignments, OSS paths, and operator dashboards are anchored on **Cleanlemons IDs**.

**Coliving (or any external system)** links via **junction / external reference** (e.g. `coliving_property_id` → `cln_property.id`), not by reusing Coliving `propertydetail` as the only source of truth inside Cleanlemons.

---

## 2) Owner: independent entity + junction when coming from Coliving

- **Owner is its own principal** in Cleanlemons (separate table / identity), same spirit as “Cleanlemons has its own clients”.
- If the commercial **client** (operator tenant) originated from Coliving integration, use a **junction** (or link table) to tie:
  - Cleanlemons `client` ↔ Coliving operator identity / external ids (TBD),
  - and **owner** ↔ property ↔ operator access as product rules require.

### Example (Coliving operator A)

- In Coliving, **operator A** has **5 properties**.
- After integration/sync to Cleanlemons:
  - **Operator A = one Cleanlemons `client`**.
  - Those **5** rows exist as **5 Cleanlemons properties** (via sync + junction to Coliving ids).
- Operator A may still **create additional properties** on the **Cleanlemons client** UI.
- Those properties (including the 5) can be **assigned or exposed** to **different Cleanlemons operators** (e.g. X, Y) according to rules configured in the **Cleanlemons client dashboard** (“operator DB” = distinct operator records under that client).

---

## 3) Integration authorisation: two supported patterns

Both end states include a **secret (or API key) generated by Cleanlemons** and stored server-side for API calls. UX can differ.

### Pattern A — “Login then Allow” (OAuth-like)

1. Coliving operator uses the **same email** in both products (e.g. `abc@gmail.com` in Coliving and as the Cleanlemons client contact / login).
2. User **signs in to Cleanlemons** with that email, then clicks **Allow** / connect (similar to Google consent).
3. **Backend** still **creates and stores** the **secret key** (generated by Cleanlemons); Coliving stores what it needs to call the API (or exchanges code for credentials — flow TBD).

### Pattern B — Manual secret in Coliving settings

1. User opens **Cleanlemons Settings**, copies **secret key** (generated by Cleanlemons).
2. Pastes into **Coliving integration** UI.

**Note:** Pattern A improves trust and email alignment; Pattern B is simpler to ship first. They can coexist.

---

## 4) Unchanged hard rules

See [integration-principles.md](./integration-principles.md):

- **One Coliving operator = one Cleanlemons client**; **per-client secret** (or per-client credential set).
- **No operator ↔ operator integration** in the current phase.
